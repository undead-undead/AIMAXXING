# AIMAXXING Security Module Documentation

The `security` module is the defensive boundary of the AIMAXXING framework. It implements a multi-layered security model to ensure that AI-generated code and external skills execute in a safe, isolated environment.

## 🛠 Technology Stack & Dependencies

- **Linux Sandboxing**: `bubblewrap` (bwrap) for unprivileged containerization.
- **macOS Sandboxing**: `sandbox-exec` (Seatbelt) for kernel-level policy enforcement.
- **Windows Sandboxing**: Windows Job Objects for process-level constraints and UI restrictions.
- **Regex Engine**: `regex` and `aho-corasick` for high-performance pattern matching in leak detection.
- **Resource Management**: `sysinfo` for monitoring memory and CPU usage within sandboxes.

## 📂 Architecture & Modules

### 1. Sandboxing & Isolation
- **`sandbox/`**: Contains OS-specific implementations for process isolation (Linux, macOS, Windows).
- **`shell_firewall.rs`**: An intent-analyzer that preemptively blocks malicious or catastrophic shell commands before they reach the OS.

### 2. Information Protection
- **`leaks/`**: Implements the `LeakDetector`. Scans stdout/stderr for API keys, passwords, and private keys using a large library of credential patterns.
- **`output_auditor.rs`**: Orchestrates the scanning and redaction process for all agent-generated content.

### 3. Application Security
- **`injection/`**: Detects potential prompt injection attempts and applies defensive markers to user inputs.
- **`vessel.rs`**: Inspects `.vessel` skill packages for binary integrity and security policy violations.
- **`skill_verifier.rs`**: Ensures that external skills follow the required security manifest before execution.

### 4. API Access Control
- **`ApiGuard` Middleware**: Implements OS-level trust logic. It distinguishes between `localhost` (fully trusted) and `remote` (X-API-Key required).
- **Static Key Exchange**: Uses a vault-stored internal key for non-local synchronization, preventing "port scanning" attacks in bridged network setpus.

## 🚀 Purpose

The `security` module's primary mission is to enable **Zero-Trust Execution**. By assuming that any code generated by an LLM is potentially malicious, it provides the "hard shell" around the `brain`, protecting the host system from data theft, file system corruption, and unauthorized network access.
