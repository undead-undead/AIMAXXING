# AIMAXXING Core Module Documentation

The `brain` module serves as the central nervous system of the AIMAXXING AI Agent framework. It provides the core abstractions, executing environments, security boundaries, and basic logic necessary to run intelligent agents.

This directory (`/home/biubiuboy/aimaxxing/core`) contains the fundamental building blocks of the framework, completely decoupled from the HTTP server mapping or the user interface layer.

## 🛠 Technology Stack & Dependencies

The `brain` module is built 100% in Rust and relies on a high-performance stack for concurrency, serialization, and sandboxed execution:

- **Async Runtime**: `tokio`, `futures`, `async-trait` for high-concurrency event loops.
- **Data & Concurrency**: `dashmap`, `parking_lot`, `rayon` (SIMD and multi-threading for data processing).
- **Serialization**: `serde`, `serde_json`, `bincode`, `schemars` (for dynamically generating JSON Schemas for AI function calling/Tools).
- **Embedded Database**: `redb` (a pure-Rust, mmap-based embedded Key-Value database for Cron jobs and persistence) and `rusqlite` (embedded SQLite).
- **Execution Sandboxes**: 
  - `wasmtime` / `wasmtime-wasi` (WebAssembly sandboxing for executing untrusted Agent skills locally).
  - `rquickjs` (QuickJS engine for running pure JavaScript/TypeScript logic directly in the Rust process).
- **Security & Networking**: `reqwest` (HTTP requests), `oauth2` / `openidconnect` (identity and MCP integrations), `argon2` (hashing).
- **Automation**: `headless_chrome` (for executing browser-based skills, taking screenshots, and web scraping).
- **Task Scheduling**: `tokio-cron-scheduler` (for background recurring tasks).
- **Telemetry**: `tracing`, `tracing-subscriber` (for granular logging and application profiling).

## 📂 Core Architecture & Modules

The source code (`core/src/`) is logically partitioned into several key domains:

### 1. `agent/` (Agent System)
The heart of the framework. Contains the abstractions for `Agent`, `AgentBuilder`, and `AgentConfig`. 
- **`message.rs`**: Defines the `Message`, `Role`, and `Content` structs used universally to communicate with LLMs.
- **`provider/`**: Interfaces defining how the agent interacts with foundation models, ensuring compatibility with LLMs like OpenAI, Anthropic, Gemini, etc. (Actual implementation sits in the `providers` crate, but the traits live here).
- **`cache.rs` / `memory/`**: Implementation of Context Window sliding, short-term conversational cache, and semantic state.

### 2. `skills/` (Tools & Sandbox Execution)
Defines what the Agent is capable of doing into the real world.
- **`tool/`**: The standard interface (`Tool`, `ToolDefinition`) used to expose Rust functions to the LLM (Function Calling). Includes built-in tools like FS operations, CLI executions, and Git automation.
- **`sandbox.rs`**: The implementation of safe execution environments. Spawns isolated processes or Wasm isolated runtimes to run arbitrary code generated by the agent without compromising the host machine.
- **`runtime/`**: Language-specific runtime bridges (Python via virtual environments, JS via QuickJS).

### 3. `knowledge/` & `store/` (RAG & Memory)
The components responsible for long-term memory and knowledge retrieval.
- **`rag.rs`**: Interfaces with the `engram` crate. Implements document chunking, semantic search, and prompt injection for RAG (Retrieval-Augmented Generation).
- **`virtual_path.rs`**: Simulates a virtual file system to map stored knowledge chunks logically so the agent can traverse them like standard files.

### 4. `security/` (Safety & Permissions)
- **`vessel.rs`**: The "Vessel" (Soul) boundary implementation. Limits the memory scope, disk space, and networking capabilities of a running agent instance.
- **Firewalls & Leak Detectors**: Analyzes outgoing Tool/API requests from the agent to detect if private keys or forbidden host paths are being accidentally or maliciously leaked.

### 5. `bus/` & `mcp/` (Eventing & Connectivity)
- **`bus/`**: An asynchronous message bus (Pub/Sub) used internally to decouple framework events (e.g., when an agent finishes an action, it fires an event to update metrics or the Panel UI).
- **`mcp/`**: Implements the **Model Context Protocol (MCP)**, allowing AIMAXXING agents to connect to standard remote MCP servers to consume external data sources and capabilities in a standardized way.

### 6. `auth/` & `session/` (Multi-Tenancy)
Handles authentication, identity validation, and user session management. Ensures that a multi-agent swarm operates distinctly across different user sessions or project boundaries.

## 🚀 What Does `brain` Do?

If you were to compile only the `brain` crate, you would not get a web server or a graphical interface. Instead, you get a highly optimized Rust library capable of:

1. **Bootstrapping an AI**: Loading an identity (Soul/Persona), mounting tools, and initializing memory.
2. **Orchestrating Thought Loops**: Receiving a user prompt, passing it to an LLM provider, parsing the intention, deciding whether to call a tool, executing that tool within a WebAssembly sandbox, collecting the result, and feeding it back into the model.
3. **Preventing Disasters**: Halting execution if the LLM decides to run `rm -rf /` by intercepting the intent via the security firewall.
4. **Scheduling**: Spawning asynchronous recurring tasks (e.g., "Check crypto prices every 5 minutes and summarize the changes").

By segregating this logic into `core` (`brain`), AIMAXXING ensures that its foundational AI behavior remains portable, lightweight, and rigorously testable (independent of the axum Gateway API or the eframe UI Panel).
