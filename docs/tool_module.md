# AIMAXXING Skills & Tools: Deep Dive Technical Documentation

This document provides a comprehensive, technically deep analysis of every built-in Tool (Skill) implemented within the `/home/biubiuboy/aimaxxing/builtin-tools/src/tool` directory. 

These tools are the effectors of the AIMAXXING framework, bridging the gap between LLM reasoning and real-world execution. Every tool implements the `Tool` async trait, utilizes `schemars` for automated JSON Schema sync, and is guarded by the execution `SecurityManager`.

---

## 1. 💻 Execution & Orchestration

### `code_interpreter.rs` (CodeInterpreterTool)
**Technical Overview**: Safely executes dynamic code payloads (Python, Javascript/Typescript) generated by the LLM.
**Implementation Details**:
- **Execution Paths**: Differentiates between lightweight tasks and heavy data engineering. For Javascript, it invokes the embedded, zero-overhead **QuickJS (`rquickjs`)** sandbox running strictly in-process within a memory-limited context. For Python, it passes the script to the `NativeShellRuntime`.
- **Environment Isolation**: Interfaces with `env_manager.rs` to trigger `uv` or `pixi` on the fly. If the LLM imports `pandas` or `numpy`, the interpreter dynamically provisions an isolated virtual environment (`~/.aimaxxing/venvs/`) before execution, preventing host OS corruption.
- **Output Redirection**: Captures `stdout` and `stderr` streams, pumping them through the globally instantiated `LeakDetector` (using regex arrays for PII/Secrets) before appending the result back to the LLM's context window.

### `forge.rs` (ForgeSkill)
**Technical Overview**: A meta-programming compiler allowing the Agent to dynamically expand its own capabilities via runtime Rust compilation.
**Implementation Details**:
- **Workflow**: The LLM writes raw `.rs` code implementing a new Tool trait. The `ForgeSkill` writes this to a temporary Cargo workspace.
- **Compilation**: Executes `cargo build --release --target wasm32-wasi` (or as a dylib) under the hood. 
- **Hot-loading**: Dynamically loads the newly compiled Wasm module into the `wasmtime` engine, linking it back into the Agent's `ToolSet`. It fundamentally enables a recursive, self-improving agentic system.

### `delegation.rs` (DelegateTool)
**Technical Overview**: The backbone of multi-agent swarm architecture.
**Implementation Details**:
- **Mechanism**: When a generic "Router" agent receives a complex prompt, it calls this tool, specifying a target `Persona` (e.g., "Senior Data Analyst").
- **State Pause**: The current agent runtime yields execution synchronously. The `DelegateTool` spawns a brand new `Agent` instance loaded with the requested persona's system prompt and targeted tools.
- **Context Passing**: It serializes the exact subset of the conversation history required, passes it to the sub-agent, awaits the Future resolution, and returns the sub-agent's summarized conclusion back to the parent agent.

### `handover.rs` (HandoverTool)
**Technical Overview**: Safely yields the infinite autonomous loop back to the human user.
**Implementation Details**:
- **Interrupt Signal**: Emits a strongly-typed `HandoverEvent` down the internal `bus/`. 
- **Functionality**: Instructs the overarching execution orchestrator that the agent requires human clarification, approval, or that the current multi-step subtask is fully completed. Prevents LLMs from infinitely looping token generation when stuck.

## 2. 🌐 Web & Network Integration

### `browser.rs` (BrowserTool)
**Technical Overview**: Heavyweight headless browser integration for navigating complex Single Page Applications (SPAs).
**Implementation Details**:
- **Library**: Wraps `headless_chrome` (a Rust port of Puppeteer).
- **Mechanism**: Spawns an isolated Chromium process. Establishes a Chrome DevTools Protocol (CDP) WebSocket connection to dictate commands natively.
- **Capabilities**: Translates LLM JSON arguments into CDP commands: `evaluate_script`, `click_element(selector)`, `type_text`, and `capture_screenshot()`. It pauses for network idle events, allowing the LLM to successfully bypass JS-rendered walls that normal HTTP clients cannot parse.

### `web_fetch.rs` (WebFetchTool)
**Technical Overview**: Ultra-fast, lightweight HTTP scraper and HTML-to-Markdown distillator.
**Implementation Details**:
- **Mechanism**: Utilizes `reqwest` to blindly grab raw HTML payloads. 
- **DOM Translation**: Uses a tree-parsing readability algorithm (simulating Mozilla's Readability) to strip out noisy semantic tags (`<nav>`, `<footer>`, `<script>`, CSS). 
- **Token Efficiency**: It converts the remaining `DOM` into clean Markdown. This ensures the token count absorbed into the LLM's context window is strictly minimized while retaining absolute data meaning.

### `web_search.rs` (WebSearchTool)
**Technical Overview**: Standardized Search Engine Results Page (SERP) bridge.
**Implementation Details**:
- **Routing**: Abstracts away the specific search provider (DuckDuckGo, Google Custom Search, Tavily). 
- **Format**: Takes the LLM's keyword query, triggers parallel asynchronous requests, and concatenates the JSON responses into a condensed `{ title, snippet, link }` interface. This allows the Agent to fetch real-time worldly context before making decisions.

## 3. 📂 Local OS & Filesystem

These tools interact with the host OS, strictly governed by the `VesselInspector` security policies.

### `filesystem.rs` (ListDir, ReadFile, WriteFile, EditFile)
**Technical Overview**: Foundational filesystem CRUD operations.
**Implementation Details**:
- **Path Canonicalization**: Before any `std::fs` call, paths are resolved using `fs::canonicalize` and strictly verified against the `base_dir` sandbox workspace path constraint. Path traversal attacks (`../../etc/passwd`) are hard-blocked by the runtime.
- **Incremental Editing**: `EditFileTool` implements line-replacement chunking (similar to unified diffs) instead of requiring the LLM to rewrite a 1000-line file completely, drastically saving output tokens and lowering latency.

### `git_ops.rs` (GitOpsTool)
**Technical Overview**: Subprocess bridge to the local `git` binary for repository management.
**Implementation Details**:
- **Command Whitelisting**: Instead of letting the LLM run arbitrary `bash` commands (which might delete branches or force push maliciously), this exposes strict JSON schemas for allowed verbs (`clone`, `commit`, `status`, `pull`, `branch`).
- **Parsing**: Translates the messy stdout of `git status` into easily digestible parsed data structures for the Agent to deduce its current state within a repo.

### `cron.rs` (CronTool)
**Technical Overview**: Proactive time-based scheduling interface.
**Implementation Details**:
- **Library**: Tightly coupled with `tokio-cron-scheduler`.
- **Mechanism**: The agent provides a cron-expression string and a payload instruction. The tool writes this to the embedded `redb` database. The background infra loop parses this DB; when the time triggers, a headless agent is autonomously spawned to execute the stored instruction.

## 4. 🧠 Memory & Context (RAG)

These tools act as the direct API client interface communicating with the high-performance `engram` module.

### `memory.rs` (FetchDocument, RememberThis, TieredSearch, SearchHistory)
**Technical Overview**: The active RAG (Retrieval-Augmented Generation) subsystem.
**Implementation Details**:
- **`RememberThis`**: Bypasses passive auto-memory. When the LLM decides a fact is critical permanent knowledge ("My user is allergic to peanuts"), it uses this tool. The text is immediately chunked, embedded via local Candle models, and slammed into the **Warm/Full** tier of the HNSW index in `engram`.
- **`TieredSearch` / `FetchDocument`**: Exposes the dual-modality search engine. Translates natural language inquiries into dense vectors + BM25 keywords, hitting the `rrf.rs` (Reciprocal Rank Fusion) engine, and parses the highly-ranked context chunks back into the chat sequence.

## 5. 🛠 Auxiliary Processing & Output

Tools designed to handle specific binary data or external service workflows that LLMs cannot synthesize natively via text.

### `chart.rs` (ChartTool)
**Technical Overview**: Generates graphical visualizations dynamically.
**Implementation Details**:
- **Library**: Uses the Rust `plotters` library.
- **Mechanism**: The Agent provides pure JSON representation of data (e.g. `[{"x": "Jan", "y": 100}]`) and a chart type enum (Line, Bar). The tool renders this data onto a pixel bitmap buffer `image::DynamicImage`, saves it to disk, and optionally returns a base64 string or file URI so the user can see the graph in the Panel UI.

### `cipher.rs` (CipherTool)
**Technical Overview**: Host-level cryptographic transformations.
**Implementation Details**:
- Saves the LLM from trying (and failing) to calculate hashes itself. Uses pure Rust crypto libraries (`sha2`, `hmac`, `aes`) to perform encrypt/decrypt, base64 operations, and hash generations. 

### `data_transform.rs` (DataTransformTool)
**Technical Overview**: High-speed, token-less data munging.
**Implementation Details**:
- **Mechanism**: If an agent needs to convert a 10MB CSV file into JSON, feeding it through the LLM is impossible (token limits) and expensive. This tool accepts a source file path and a target format. It uses Rust's `serde` (`serde_json`, `csv`, `quick-xml`) to flawlessly convert formats natively in milliseconds, saving the result to a new file.

### `image.rs` (ImageProcessingTool)
**Technical Overview**: Pixel manipulation pipeline.
**Implementation Details**:
- **Library**: Relies on the standard `image` crate. 
- **Mechanism**: Executes crop, resize, and format conversions locally. Prepares images for vision-model ingestion (downsampling huge JPEGs to fit within the `gpt-4o` token payload constraints).

### `mailer.rs` (MailerTool)
**Technical Overview**: Native SMTP networking.
**Implementation Details**:
- **Library**: `lettre` for safe email construction.
- **Mechanism**: Uses Vault-injected credentials to assemble MIME emails (HTML + Text fallbacks) and securely dispatches them to external servers on behalf of the agent, handling TLS handshakes directly in Rust.

### `notifier.rs` (NotifierTool)
**Technical Overview**: Cross-platform system alerting.
**Implementation Details**:
- Utilizes `notify-rust` (DBus/Win32/NSUserNotification) to push UI toast notifications to the Host OS, or fires HTTP POST Webhooks to external endpoints (Slack/Discord) when a long-running sub-agent finishes an autonomous compilation loop.

### `text_extract.rs` (TextExtractTool)
**Technical Overview**: Universal document ingestion pipeline.
**Implementation Details**:
- **Format Support**: Parses binary PDF (`lopdf`), DOCX / PPTX (`dotx`), and EPUB files natively without external dependencies. 
- **Mechanism**: Strips out images and styling, returning raw contiguous text buffers. These buffers are then usually piped directly into `memory.rs` by the agent to be chunked into the Vector DB.

### `voice.rs` (TranscribeTool & SpeakTool)
**Technical Overview**: Audio processing bridge.
**Implementation Details**:
- **Mechanism (STT)**: Accepts `.wav` or `.mp3` paths. Either routes them to OpenAI's Whisper API via `reqwest`, or (if configured locally) pipes them to `candle-transformers` whisper models running on local CPU/GPU to generate text transcripts.
- **Mechanism (TTS)**: Flips text into audio streams using generic TTS APIs, saving the buffer to the sandboxed filesystem for the frontend to play.
